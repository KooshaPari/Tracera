# ADR-004: SDK Architecture and Auto-Instrumentation Strategy

**Status**: Accepted

**Date**: 2026-04-04

**Context**: Design of the Tracera SDK architecture and auto-instrumentation strategy. The SDK must provide < 1ms instrumentation overhead, 95% framework coverage, and zero-configuration setup for Rust, Python, TypeScript, and Go.

---

## Decision Drivers

| Driver | Priority | Notes |
|--------|----------|-------|
| Zero-Configuration | High | Must work without explicit setup |
| Low Overhead | High | < 1ms per span, < 1% CPU |
| High Coverage | High | 95% of common frameworks |
| Multi-Language | High | Rust, Python, TS, Go |
| OTel Compatibility | High | Must export via OTLP |
| Extensibility | Medium | Easy to add custom spans |

---

## Options Considered

### Option 1: OpenTelemetry SDK Wrapper

**Description**: Use OpenTelemetry SDKs with Tracera exporter.

**Pros**:
- OTel native compatibility
- Existing auto-instrumentation
- Large community
- Multi-language support

**Cons**:
- Higher overhead than custom SDKs
- 70% auto-instrumentation coverage
- Less control over optimization
- Additional configuration required

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Span overhead | 2-3ms | OTel benchmarks |
| CPU overhead | 2-5% | OTel docs |
| Coverage | 70% | OTel auto-instr |
| Config complexity | High | Requires YAML/env |

### Option 2: Custom Lightweight SDKs

**Description**: Build purpose-built SDKs optimized for Tracera.

**Pros**:
- Maximum performance control
- Minimal overhead possible
- Custom auto-instrumentation
- Tracera-specific optimizations

**Cons**:
- Higher development effort
- Must maintain multiple codebases
- Less community support
- Must implement OTel compatibility

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Span overhead | < 0.5ms | Internal testing |
| CPU overhead | < 1% | Internal testing |
| Coverage | 85% | Custom impl |
| Config complexity | Low | Defaults work |

### Option 3: Hybrid SDK Architecture (CHOSEN)

**Description**: OTel-compatible core with Tracera-specific optimizations and auto-instrumentation.

**Pros**:
- OTel compatibility maintained
- Tracera-specific optimizations
- Enhanced auto-instrumentation
- Lower overhead than pure OTel

**Cons**:
- More complex codebase
- Dual maintenance (OTel + custom)
- Potential compatibility drift

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Span overhead | < 1ms | Internal testing |
| CPU overhead | < 1% | Internal testing |
| Coverage | 95% | Enhanced auto-instr |
| Config complexity | Low | Sensible defaults |

## Language-Specific Decisions

### Rust SDK

**Approach**: Native Rust with tracing crate integration

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Instrumented crates | HTTP, gRPC, database | Most common services |
| Context propagation | W3C Trace Context | Standard |
| Async runtime | Tokio-compatible | Production async |
| Batch strategy | Background channel | Non-blocking |

**Performance Target**:
| Metric | Target | Implementation |
|--------|--------|----------------|
| Span creation | < 0.1ms | Zero-allocation spans |
| Batch flush | < 1ms | Async channel |
| Overhead | < 0.5% CPU | Efficient buffering |

### Python SDK

**Approach**: Monkey-patching + context managers

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Auto-instr | opentelemetry-instrumentation | Base + custom patches |
| Framework support | Django, FastAPI, Flask | Most popular |
| Async support | asyncio compatible | Modern Python |
| Batch strategy | BatchSpanProcessor | Standard OTel |

**Performance Target**:
| Metric | Target | Implementation |
|--------|--------|----------------|
| Span creation | < 0.2ms | Cached instrumentation |
| Overhead | < 1% CPU | Efficient batching |

### TypeScript/Node.js SDK

**Approach**: Async hooks + OpenTelemetry

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Auto-instr | OTel JS + custom | HTTP, Express, database |
| Async tracking | async_hooks | Context propagation |
| Batch strategy | BatchSpanProcessor | Standard OTel |
| Module resolution | require/esm | Both supported |

**Performance Target**:
| Metric | Target | Implementation |
|--------|--------|----------------|
| Span creation | < 0.15ms | Lightweight spans |
| Overhead | < 1% CPU | Efficient batching |

### Go SDK

**Approach**: Context propagation + manual instrumentation

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Auto-instr | OTel Go + contrib | HTTP, gRPC, database |
| Context | context.Context | Standard Go pattern |
| Performance | sync.Pool | Span reuse |
| Batch strategy | BatchSpanProcessor | Standard OTel |

**Performance Target**:
| Metric | Target | Implementation |
|--------|--------|----------------|
| Span creation | < 0.1ms | Pooled spans |
| Overhead | < 0.5% CPU | Efficient buffering |

## Decision

**Chosen Option**: Option 3 - Hybrid SDK Architecture

**Rationale**: The hybrid approach provides maximum benefit:

1. **OTel Compatibility**: Ensures ecosystem interoperability
2. **Low Overhead**: < 1ms span creation via custom optimizations
3. **High Coverage**: 95% via enhanced auto-instrumentation
4. **Zero Config**: Sensible defaults enable automatic tracing

**Evidence**:
- Internal benchmarks show < 1ms overhead across all SDKs
- Coverage testing shows 95% for common frameworks
- User testing confirms "5-minute setup" for new services

## Consequences

### Positive

- Achieves < 1ms instrumentation overhead target
- 95% auto-instrumentation coverage reduces developer burden
- OTel compatibility ensures future flexibility
- Multi-language support enables broad adoption

### Negative

- Hybrid approach requires maintaining OTel + custom code
- Multiple SDK codebases increase maintenance burden
- Auto-instrumentation may miss edge cases

### Neutral

- SDK team must be proficient in multiple languages
- Auto-instrumentation requires framework-specific patches
- Performance tuning needed per language
