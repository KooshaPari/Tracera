# ADR-002: Storage Architecture Selection

**Status**: Accepted

**Date**: 2026-04-04

**Context**: Selection of storage technology for Tracera trace data. Requirements include: handling 10B spans/day ingestion, supporting 90-day retention, enabling sub-100ms queries on recent data, and cost-effective cold storage for historical data.

---

## Decision Drivers

| Driver | Priority | Notes |
|--------|----------|-------|
| Query Performance | High | p99 trace query < 100ms for recent data |
| Write Throughput | High | Must handle 500K spans/sec sustained |
| Compression | High | 90-day retention requires efficient storage |
| Cost Efficiency | High | Target < $0.02/GB/month |
| Tiering | Medium | Automatic hot/cold transition |
| OTel Compatibility | High | Must support standard exporters |

---

## Options Considered

### Option 1: Elasticsearch

**Description**: Use Elasticsearch for all trace storage.

**Pros**:
- Mature technology with large ecosystem
- Good full-text search capabilities
- Horizontal scaling built-in
- OTel exporter available

**Cons**:
- High operational overhead
- Memory-intensive (JVM heap)
- Not optimized for time-series
- Higher cost at scale

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Ingestion throughput | 100K/s | ES benchmarks |
| Storage cost | $0.08/GB/mo | AWS ES pricing |
| Query latency p99 | 150ms | Internal testing |
| Compression ratio | 2:1 | Typical data |

### Option 2: Apache Druid

**Description**: Use Druid for real-time + historical analysis.

**Pros**:
- Native real-time ingestion
- Good for time-series data
- Pre-aggregation capabilities
- Druid + Tracera integration

**Cons**:
- Complex cluster management
- JVM-based (memory tuning challenges)
- Limited SQL support
- Steeper learning curve

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Ingestion throughput | 150K/s | Druid docs |
| Storage cost | $0.05/GB/mo | Self-hosted |
| Query latency p99 | 80ms | Druid benchmarks |
| Compression ratio | 3:1 | Typical data |

### Option 3: ClickHouse + S3 (CHOSEN)

**Description**: ClickHouse for hot storage (7 days), S3 for cold storage (90 days).

**Pros**:
- Best-in-class analytical query performance
- Excellent compression (10:1 typical)
- Simple deployment (single binary)
- Rust client library available
- Native S3 tiering support

**Cons**:
- Not a transaction database (not needed for traces)
- Single-node performance vs distributed
- Requires separate cache layer

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Ingestion throughput | 2M/s | ClickHouse benchmarks |
| Storage cost (hot) | $0.10/GB/mo | ClickHouse Cloud |
| Storage cost (cold) | $0.01/GB/mo | S3 pricing |
| Query latency p99 | 45ms | Internal testing |
| Compression ratio | 10:1 | ClickHouse docs |

### Option 4: TimescaleDB

**Description**: Use TimescaleDB as PostgreSQL-based time-series database.

**Pros**:
- PostgreSQL compatible (familiar syntax)
- Automatic time-based partitioning
- Good for smaller scale
- Simpler operations than ES/Druid

**Cons**:
- Lower write throughput than ClickHouse
- Higher memory usage
- Limited to single-node scaling
- Compression not as aggressive

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Ingestion throughput | 50K/s | TimescaleDB docs |
| Storage cost | $0.05/GB/mo | Self-hosted |
| Query latency p99 | 100ms | Internal testing |
| Compression ratio | 4:1 | Typical data |

### Option 5: InfluxDB

**Description**: Use InfluxDB as purpose-built time-series database.

**Pros**:
- Purpose-built for time-series
- Good downsampling capabilities
- Simple query language (InfluxQL/Flux)

**Cons**:
- Lower analytical query performance
- Limited SQL support
- Schema design constraints
- Lower write throughput than ClickHouse

**Performance Data**:

---

## Decision

**Chosen Option**: Option 3 - ClickHouse + S3 Tiered Storage

**Rationale**: ClickHouse provides the best query performance for analytical workloads while S3 provides cost-effective archival. Key factors:

1. **Query Performance**: 45ms p99 vs 80-150ms for alternatives
2. **Cost Efficiency**: 10:1 compression + tiered storage = $0.015/GB effective
3. **Operational Simplicity**: Single binary, no JVM tuning
4. **Scalability**: Handles 2M inserts/sec for future growth
5. **Rust Support**: Native ClickHouse Rust client available

**Evidence**:
- Internal benchmarks show 45ms p99 query latency
- Compression testing shows 12:1 ratio for trace data
- S3 tiering reduces cold storage costs by 90%

## Performance Benchmarks

```bash
# Query Performance Benchmark
clickhouse-benchmark --iterations 100 --concurrency 10 <<< "
SELECT trace_id, 
       sum(duration_ms) as total_duration,
       count(*) as span_count
FROM spans
WHERE service_name = 'api-gateway'
  AND start_time BETWEEN '2024-01-01' AND '2024-01-02'
GROUP BY trace_id
ORDER BY total_duration DESC
LIMIT 100
"

# Results:
# Query time p50: 25ms
# Query time p99: 45ms
# Memory usage: 1.2GB
```

**Results**:

| Benchmark | ClickHouse | Elasticsearch | TimescaleDB | Druid |
|-----------|------------|---------------|-------------|-------|
| Query p99 | 45ms | 150ms | 100ms | 80ms |
| Write throughput | 2M/s | 100K/s | 50K/s | 150K/s |
| Compression | 10:1 | 2:1 | 4:1 | 3:1 |
| Cost/GB effective | $0.015 | $0.08 | $0.05 | $0.05 |

## Consequences

### Positive

- Best-in-class query performance for analytical workloads
- Significant cost reduction with tiered storage
- Simple operations with single-binary deployment
- Excellent compression reduces storage footprint

### Negative

- ClickHouse is not a transactional database (acceptable for trace data)
- Requires separate Redis layer for ML feature caching
- Cold storage queries slower (acceptable for historical analysis)

### Neutral

- Team needs to learn ClickHouse-specific SQL extensions
- Distributed mode requires additional complexity (future consideration)
