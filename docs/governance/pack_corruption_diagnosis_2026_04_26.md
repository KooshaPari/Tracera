### What works
- HEAD branch and current working tree are fully resolvable.
- `git log` on the current HEAD lineage works.
- All 632 reachable-via-rev-list commits enumerate without fatals.

### What is broken
- 319 historical tree links cannot be followed. Cannot traverse arbitrary historical paths
  (e.g. the `proposals/*` query that triggered this).
- The reported `dab120b1165e0aeb6eb410f75c3ab69483e485d69` is one of the 106 missing trees;
  parent tree `b668e0c818dbc183c33fa981051a6ffe467c3ced` references it but cannot read it.
- `tmp_pack_*` files (46 of them, ~4.8 GB) indicate **interrupted gc / repack operations** —
  consistent with prior memory note "disk too tight, gc deferred."
- commit-graph file references 751 commits no longer in the object DB → corrupt commit-graph
  must be regenerated.

## Root cause hypothesis

Sequence (matches W-78D family + pruner-atime limitation memory):
1. Multiple concurrent agents ran `git fetch` / `git gc --auto` against this canonical.
2. Disk pressure (currently 18 Gi free, 98% full) interrupted repack mid-flight, leaving
   `tmp_pack_*` blobs but no matching `.idx`.
3. Subsequent gc runs pruned objects whose only copy lived in those interrupted temp packs,
   resulting in genuinely missing reachable trees/blobs.
4. commit-graph was written before the loss and now points at vanished commits.

## Recommended fix sequence (USER-EXECUTED, in order)

DO NOT run any of these without confirming current disk state and pausing concurrent agents.

### Step 0 — Pre-flight (mandatory)
```bash
df -h /Users/kooshapari/CodeProjects/Phenotype/repos   # need >= 30 Gi free
# Pause all background agents touching this repo. Confirm no running processes:
lsof +D /Users/kooshapari/CodeProjects/Phenotype/repos/.git 2>/dev/null | head
```

### Step 1 — Free disk first (use target-pruner per memory)
```bash
cd /Users/kooshapari/CodeProjects/Phenotype/FocalPoint
./target/release/target-pruner --prune
# If still tight, manually `rm -rf <repo>/target` for non-active repos
```

### Step 2 — Remove garbage temp packs (safe; they are not indexed)
```bash
cd /Users/kooshapari/CodeProjects/Phenotype/repos
rm -f .git/objects/pack/tmp_pack_*    # ~4.8 GB reclaim
```

### Step 3 — Drop poisoned commit-graph (regenerated by gc)
```bash
rm -f .git/objects/info/commit-graph
rm -rf .git/objects/info/commit-graphs
```

### Step 4 — Trim broken reflog entry
```bash
git reflog expire --expire=now --expire-unreachable=now --all
```

### Step 5 — Re-fetch missing objects from network where possible
```bash
git fetch --all --prune
git fetch origin '+refs/*:refs/remotes/origin/*' --prune  # Tracera
git fetch pheno  '+refs/*:refs/remotes/pheno/*'  --prune  # phenoShared
```
Note: dab120b1 specifically is unlikely to be re-fetched if its referencing commit is no
longer on any remote ref. That history is lost; only ref-tip-reachable history will heal.

### Step 6 — Repack and prune
```bash
git gc --aggressive --prune=now
```

### Step 7 — Verify
```bash
git fsck --full --strict 2>&1 | tail -40
git count-objects -v
```

Expected outcome after Steps 1–7:
- Missing-blob/tree counts drop sharply (anything reachable from a current remote ref heals).
- Residual missing objects = orphaned history (proposals/* old paths, deleted branches);
  these are unrecoverable but should no longer be reachable from any live ref, so fsck
  becomes clean for current branches.

### Step 8 — If Step 7 still shows reachable missing objects on live branches
Last-resort options (USER decision required):
- **a)** Drop the affected local-only branches that reach the missing objects (preserves origin-reachable history).
- **b)** Re-clone fresh from each origin into a new directory and rebuild the multi-remote agglomeration. Most invasive, but guarantees a clean object DB.
